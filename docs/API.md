## 1. 화면 ↔ API 매핑 한눈에 보기

```
[시작화면]        ① GET /participant-count   (진입 시 · 손바닥 이스터에그 시드 겸용)
    ↓ 시작하기      ③ GET /questions 프리페치  ★ 여기서 미리 받기 시작
[학교 선택]        ② GET /schools?query=      (타이핑 디바운스 · 건너뛰기 시 schoolId 0)
    ↓
[로딩 ①]          — 네트워크 없음 (③ 도착 완료 + 일러스트 프리로드)
[질문 1~8]         — 네트워크 없음. 선택 후 reactionText(반응 스토리) 노출 → 다음 질문
    ↓ 마지막 답변    ④ POST /histories        ★ 로딩 ② 연출이 응답 대기를 가림
[로딩 ②]          ④ 응답 수신 → 결과 일러스트 프리로드 후 화면 전환
[결과 카드]        — 네트워크 없음 (④ 응답으로 렌더)
    │ 공유 링크로 재진입한 사용자 → ⑤ GET /histories/{id}
    ↓ 결과 분포 확인하기
[순위]             ⑥ GET /rankings + ⑦ GET /schools/{id}/rankings  (결과 화면에서 프리페치)
```

핵심 원칙: **사용자가 "로딩을 기다리는" 순간을 없앤다.** 연출 로딩 ①·②가 모든 네트워크 대기를 가리고, 그 외 화면 전환은 즉시 이루어진다.

## 2. 확정된 명세 (백엔드 전달본)

### ① 전체 참여자 수 조회
- `GET /participant-count`
- **사용처**: 시작화면 "현재 \_\_\_명이 참여중" + 피 손바닥 이스터에그 시드.
- **호출 시점**: 시작화면 진입 시 1회. 스피너 없이 플레이스홀더 → 도착 시 채움.
- **응답**: `{ "participantCount": 357 }`
- ✅ 이전 검토에서 요청한 대로 "목록"이 아닌 "수"로 확정됨.

### ② 학교 목록 검색
- `GET /schools?query=` — ✅ 협의 확정 (query 파라미터 방식, 상세 명세 문서 수신 대기)
- **사용처**: 학교 선택 화면의 단일 검색 박스 실시간 자동완성.
- **호출 시점**: 타이핑 디바운스 150~300ms + 동일 쿼리 메모리 캐시.
- **응답 제안** (기존 파라메터명 규칙 준수):
  ```json
  { "schools": [ { "schoolId": 12, "schoolName": "인하고등학교" } ] }
  ```
- 동명 학교 구분용 지역 필드 포함 여부는 명세 수신 시 확인 (3장 R-1).

### ③ 전체 질문 및 선택지 조회
- `GET /questions`
- **사용처**: 질문 화면 8문항. **호출 시점**: ★ 시작하기 클릭 직후 프리페치.
- **응답**:
  ```json
  { "questions": [
      { "questionId": 1,
        "content": "담을 넘다가 경비실 쪽에서 인기척이 들린다",
        "imageUrl": "https://.../q1.png",
        "choices": [
          { "choiceId": 1, "choiceOrder": 1, "content": "…", "reactionText": "…" },
          { "choiceId": 2, "choiceOrder": 2, "content": "…", "reactionText": "…" }
        ] } ] }
  ```
- ✅ **`reactionText` (선택 후 반응 스토리) — 채택 확정**: 어떤 선택지를 골라도 **다음 질문은 동일한 선형 구조**이고, 선택은 결과 계산에만 반영된다. 선택 직후 해당 선택지의 반응 텍스트를 연출로 보여준 뒤 다음 질문으로 진행. (반응 연출의 비주얼 디자인은 필요 — DESIGN.md 잔여 이슈)
- 2지선다 고정과 일치. ✅

### ④ 결과 계산 및 저장 (통합)
- `POST /histories`
- **사용처**: 마지막 답변 직후 제출 → 응답이 결과 화면의 데이터 소스.
- ✅ 계산과 저장이 한 번의 호출로 통합됨 (프론트 요청 반영).
- ✅ `id` 발급 → 공유 링크 식별자 확보 (프론트 요청 반영).
- **요청**:
  ```json
  { "schoolId": 12, "choiceIds": [1, 4, 6, 8, 9, 12, 14, 16] }
  ```
- ✅ **`schoolId` 규칙 확정**: null 불허. **건너뛰기 선택 시 `0`** 을 보낸다. (0일 때 응답의 `schoolName` 값은 확인 필요 — 3장 R-2)
- ✅ **`description` = 수식어**: 결과 카드는 `description`("밤새 조잘거려서 귀찮게 하는") + `ghostName`("수다쟁이 처녀귀신") 조합으로 표시. 랭킹 화면도 동일 규칙.
- ✅ **공유 미리보기 이미지**: 별도 OG 이미지 제작 없이 `imageUrl`(유형 일러스트)을 `getTossShareLink`의 `ogImageUrl` 파라미터에 그대로 사용. 단, 이 URL은 토스 외부(카톡 등)에서 접근하므로 **인증 없는 공개 URL**이어야 함.
- **응답 (201)**:
  ```json
  { "id": 1004,
    "schoolName": "인하고등학교",
    "ghostType": "WATER_GHOST",
    "ghostName": "물귀신",
    "description": "한 줄 설명이 들어가는 공간입니다.",
    "imageUrl": "https://.../water_ghost.png",
    "aiStory": "밤마다 학교 분수대 주변을 떠도는 이 귀신은...",
    "bestMatch":  { "ghostName": "설녀",   "comment": "말 한마디로 사람 얼려버리는 설녀" },
    "worstMatch": { "ghostName": "홍길동", "comment": "맨날 나타난다고 말만 하고 안 나타나는 홍길동" },
    "createdAt": "2026-07-13T22:45:12" }
  ```

### ⑤ 결과 상세 조회 (결과창·공유 링크 공용)
- `GET /histories/{id}`
- **사용처**: 공유 링크(`/result?id=1004`) 재진입. 본인 결과는 ④ 응답을 그대로 사용 — 이 API 호출 없음.
- **응답**: ④와 동일 구조.

### ⑥ 전체 순위 조회
- `GET /rankings`
- **사용처**: 순위 화면. **호출 시점**: ★ 결과 화면 진입 시 백그라운드 프리페치.
- **응답** — ✅ 순위 화면 표시용 유형 필드 추가 확정 (기존 파라메터명 `ghostName`·`description` 그대로 사용):
  ```json
  { "participantCount": 357,
    "rankings": [
      { "rank": 1, "ghostType": "WATER_GHOST", "ghostName": "물귀신",
        "description": "한 줄 설명이 들어가는 공간입니다.",
        "count": 85, "percent": 23.8 } ] }
  ```

### ⑦ 학교별 순위 조회
- `GET /schools/{schoolId}/rankings`
- **사용처**: 순위 화면의 우리 학교 분포. ⑥과 병렬 프리페치 (학교 선택자만).
- **응답**: `{ "schoolName": "인하대학교", "participantCount": 53, "rankings": [...] }` — rankings 항목은 ⑥과 동일 구조 (`ghostName`·`description` 포함).

> **학교 간 참여 랭킹(어느 학교가 제일 많이 했나)은 MVP에서 제외하기로 결정** (2026-07-19). 순위 화면은 전체 순위 + 우리 학교 분포로 구성.

### 🟡 남은 확인 사항

| R-2 | `schoolId: 0` 응답 처리 | ④·⑤ 응답에서 건너뛰기 사용자의 `schoolName` 값 (null? "없음"?) 및 ⑦ 호출 불가 처리 |

| R-8 | 에러 규격 | 공통 에러 응답 형식·상태코드 정의 없음. ④ 실패 시 프론트는 조용히 재시도 → 2회 실패 시 재시도 버튼 (답변은 메모리 보존) |
| R-9 | 캐싱 헤더 | ③은 정적 성격 → `Cache-Control` 협의. ⑥·⑦은 서버측 단기 캐시(30초~1분)면 충분 |

## 4. 로딩 최소화 전략 (프리페치 타임라인)

| 사용자가 보는 화면 | 그 뒤에서 하는 일 |
|---|---|
| 시작화면 | ① 참여자 수 (플레이스홀더 → 채움, 스피너 없음) |
| 시작하기 클릭 순간 | ③ `GET /questions` 발사 |
| 학교 선택 (검색하는 동안) | ③ 응답 도착, 문항 일러스트 8장 프리로드 시작 |
| 로딩 ① (연출) | 일러스트 프리로드 마무리 |
| 질문 1~8 (+반응 텍스트) | 네트워크 0회 — 전부 메모리에 있음 |
| 마지막 답변 클릭 순간 | ④ `POST /histories` 발사 |
| 로딩 ② (연출) | ④ 응답 수신 → 결과 일러스트(`imageUrl`) 프리로드 → 완료 후 전환 |
| 결과 카드 | 네트워크 0회. 진입과 동시에 ⑥·⑦ 백그라운드 프리페치 |
| 순위 | 네트워크 0회 — 이미 도착해 있음 |

- **연출 로딩 = 레이턴시 마스크**: "최소 노출 시간"과 "데이터 준비 완료" 중 늦은 쪽까지만 유지.
- **이미지가 진짜 병목**: 결과 화면 전환은 결과 일러스트 `onload` 이후에 실행.
- **학교 검색 캐시**: 동일 쿼리 메모리 캐시 + 이전 결과 유지(stale-while-revalidate)로 목록 깜빡임 방지. 디바운스 150~300ms.
- **reactionText 채택 시**: 반응 연출 시간(1~2초)이 자연스러운 페이스 조절 겸 이미지 프리로드 버퍼가 됨 — 로딩 체감엔 오히려 유리.
- **번들**: 순위 등 후순위 라우트 코드 스플리팅 (현재 번들 1.2MB 경고와 연결).
