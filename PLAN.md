## 1. 개요

| 항목 | 내용 |
|---|---|
| 프로젝트명 | 귀신 유형 테스트 (가칭) |
| 한 줄 소개 | 8개의 질문으로 알아보는 나의 귀신 유형 — 오싹한 여름 납량 심리테스트 |
| 출품 대상 | 토스 여름 챌린지 |
| 플랫폼 | 앱인토스(Apps in Toss) 미니앱 — WebView 방식 |
| 담당 범위 | 이 레포는 **프론트엔드(React)**. 유형 판정 로직·통계는 백엔드 담당 |
| 테스트 구조 | 8문항 → 16개 귀신 유형 판정 |

### 왜 이 아이템인가
- 여름 = 납량 시즌. 계절성과 챌린지 시기가 정확히 맞물림.
- 유형 테스트는 "내 결과 공유 → 친구 유입"의 바이럴 루프가 검증된 포맷.
- 학교별 통계로 또래 집단 내 경쟁·소속감 요소를 더해 확산 동력을 강화.

## 2. 컨셉 & 톤앤매너

**오싹한 호러풍.** 여름 납량특집 무드의 진지한 호러 연출을 지향한다.

- **비주얼**: 어두운 배경(딥 네이비/블랙), 낮은 채도, 붉은 포인트 컬러. 안개·촛불·글리치 같은 공포 연출 모티프.
- **연출**: 화면 전환 시 페이드/글리치 효과, 질문 진행마다 미묘하게 어두워지는 배경 등 몰입감 있는 인터랙션. (성능·심사 기준 내에서)
- **카피 톤**: 으스스하지만 불쾌하지 않게. 혐오·잔혹 묘사는 배제 (전연령 심사 통과 기준).
- **사운드**: MVP에서는 제외, 추후 결정. (미니앱 환경 자동재생 제약 확인 필요)

## 3. 타겟 & 목표

- **주 타겟**: 10~20대. 학교별 통계 기능이 있으므로 중·고·대학생이 핵심 타겟.
- **핵심 지표(가안)**: 테스트 완주율, 결과 공유율, 공유 링크 통한 재유입률, 학교별 참여자 수.

## 4. 서비스 플로우

```
[홈(인트로)]
   ↓ 시작하기
[학교 선택]  ← 검색으로 학교 선택 (테스트 시작 전 입력, 건너뛰기 가능)
   ↓
[로딩 ①]  ← "귀신을 불러들이는 중…" 연출 (질문 데이터 프리페치)
   ↓
[질문 1 ~ 8]  ← 진행률 표시, 뒤로가기 가능, 2지선다
   ↓ 8번 응답 완료
[판정 로딩]  ← 백엔드에 응답 제출, 연출용 로딩 (예: "당신에게 붙은 귀신을 찾는 중…")
   ↓
[결과 화면]
   ├─ 내 귀신 유형 (이미지 + 이름 + 설명)
   ├─ 귀신 궁합/상성 (잘 맞는 유형 / 상극 유형)
   ├─ 유형별 통계 (전체 참여자 중 내 유형 비율)
   ├─ 학교별 통계 (우리 학교 유형 분포 / 참여자 수)
   └─ 공유하기 / 다시하기
```

- 학교 선택을 시작 전에 받으므로, 학교를 선택한 사용자의 응답은 **모두 학교 통계에 집계**된다.
- **건너뛰기 허용 (확정)**: 진입 이탈을 줄이기 위해 학교 선택 없이도 테스트 진행 가능. 건너뛴 사용자는 전체 통계에만 집계되고, 결과 화면에서 학교별 통계 섹션 대신 "학교 선택하고 우리 학교 통계 보기" CTA를 노출한다.

## 5. 화면 정의

### 5.1 홈 (인트로)
- 타이틀, 메인 비주얼(호러 무드), 참여자 수 카운터(선택), [시작하기] CTA.
- 공유 링크로 유입된 경우에도 이 화면으로 랜딩.

### 5.2 학교 선택
- **검색 박스 하나만** 배치 — 사용자 피로도 최소화가 원칙. 타이핑과 동시에 실시간 자동완성 목록 표시, 탭 한 번으로 선택 완료.
- **데이터 소스 (확정)**: 백엔드가 공공데이터(전국 학교 목록) 기반으로 학교 검색 API를 제공한다.
- 하단에 "건너뛰기" 링크 제공 (§4 플로우 참고).
- 디자인 시안 없음 — 스타일 가이드 기반 프론트 자체 제작 (docs/DESIGN.md §3.5).

### 5.3 질문 (8문항)
- 한 화면에 질문 1개 + 질문별 일러스트 + 선택지 (**2지선다 고정**).
- 선택 직후 해당 선택지의 **반응 스토리 텍스트(reactionText)** 를 연출로 노출 후 다음 질문 진행 — 어떤 선택이든 다음 질문은 동일(선형 구조), 선택은 결과 계산에만 반영.
- 상단 진행률 인디케이터 (1/8 … 8/8).
- 이전 문항으로 돌아가 답 수정 가능.
- 질문·선택지 텍스트는 백엔드 API로 받는다 (프론트 하드코딩 금지 — 문항 수정 시 배포 없이 반영).

### 5.4 판정 로딩
- 최소 노출 시간을 둔 연출용 로딩 (API가 빨라도 1.5~2초 유지해 긴장감 조성).
- API 실패 시 재시도 UI.

### 5.5 결과
- **내 유형 카드**: 귀신 일러스트, 유형명, 캐치프레이즈, 상세 설명, 특징 키워드.
- **궁합/상성**: 잘 맞는 귀신 · 상극인 귀신 각 1개 이상 (백엔드 정의 데이터).
- **유형별 통계**: "전체 참여자 중 나와 같은 유형 N%" + 16유형 분포.
- **학교별 통계**: 우리 학교 참여자 수, 학교 내 유형 분포. 학교를 건너뛴 사용자에게는 학교 선택 유도 CTA 노출. (참여자 수 기준 학교 간 랭킹은 MVP 제외 — 2026-07-19 결정)
- **공유하기**: 결과 카드 이미지 + 토스 공유 링크.
- **다시하기**: 홈으로 이동.

## 6. 핵심 기능 명세

### 6.1 결과 공유
- `getTossShareLink`로 `intoss://<앱이름>/result?id=<결과ID>` 형태의 딥링크 공유 링크 생성.
- `ogImageUrl`에는 **유형 일러스트(`imageUrl`)를 그대로 재사용** — 별도 OG 이미지 제작 없음 (2026-07-19 결정). 카톡 등 외부에서 접근하므로 인증 없는 공개 URL이어야 함.
- 공유 링크로 진입한 사용자에게는 "너의 귀신도 알아봐" CTA로 테스트 시작 유도.

### 6.2 유형별 통계 (전체)
- 백엔드 집계 API 사용. 결과 화면에서 조회.
- 표시: 내 유형 비율 강조 + 전체 분포 (막대 or 리스트).

### 6.3 귀신 궁합/상성
- 유형 간 궁합 매핑은 백엔드(또는 정적 데이터)에 정의.
- 결과 화면에 "잘 맞는 귀신 / 상극인 귀신" 카드로 노출 → 친구에게 "너 해봐, 우리 궁합 보자" 공유 동기 부여.

### 6.4 학교별 통계
- 결과 화면에서 내 학교의 유형 분포·참여자 수 표시.
- ~~학교 랭킹~~ — 참여자 수 기준 학교 간 랭킹은 **MVP에서 제외** (2026-07-19 결정, 확장 여지로만 유지).

## 7. 콘텐츠 — 16개 귀신 유형

판정 로직과 유형 정의는 **백엔드에 이미 정의되어 있음.** 프론트는 아래 메타데이터를 API로 받아 렌더링한다.

유형별 필요 필드 (백엔드와 스키마 합의 필요):

| 필드 | 용도 |
|---|---|
| `id` | 유형 식별자 (딥링크 파라미터로도 사용) |
| `name` | 유형명 (예: 물귀신, 처녀귀신, 도깨비 …) |
| `catchphrase` | 결과 카드 한 줄 카피 |
| `description` | 상세 설명 |
| `keywords` | 특징 키워드 (해시태그형) |
| `imageUrl` | 일러스트 |
| `ogImageUrl` | 공유용 OG 이미지 |
| `goodMatch` / `badMatch` | 궁합/상극 유형 id |

- 판정 로직·유형 정의: 백엔드에 정의 완료.
- 일러스트 16종 + OG 이미지 16종: **디자인 팀 전달 완료.**

## 8. 기술 스택 (프론트엔드)

| 항목 | 선택 |
|---|---|
| 빌드/프레임워크 | Vite + React 18 + TypeScript (세팅 완료) |
| 미니앱 SDK | `@apps-in-toss/web-framework` (WebView 방식, 세팅 완료) |
| 디자인 시스템 | TDS Mobile (`@toss/tds-mobile` + `@toss/tds-mobile-ait`, `TDSMobileAITProvider` 적용 완료) |
| 스타일링 | `@emotion/react` (TDS 의존) + 전역 CSS (호러 테마 변수는 `src/index.css`) |
| 앱 설정 | `granite.config.ts` 생성 완료 — ⚠️ 콘솔 등록 후 `appName`/`icon` 실제 값으로 교체 필요 |
| 상태 관리 | ⚠️ 미정 (규모상 Context/zustand 수준이면 충분할 것) |

- 디렉토리 구조: `src/pages` (화면), `src/components` (공용 컴포넌트), `src/api` (백엔드 연동).
- 실기기 테스트 시 `granite.config.ts`의 `web.host`를 로컬 IP로, `dev` 커맨드를 `vite --host`로 변경 후 샌드박스 앱에서 `intoss://{appName}`으로 접근.

### 백엔드 연동 API (프론트가 기대하는 인터페이스 — 백엔드와 합의 필요)

> 상세 스펙·호출 시점·로딩 최소화 전략: [docs/API.md](./docs/API.md)

| API | 용도 |
|---|---|
| `GET /participant-count` | 전체 참여자 수 (시작화면 카운터) |
| `GET /schools?query=` | 학교 검색 (공공데이터 기반, 백엔드 제공) |
| `GET /questions` | 8문항 + 선택지 + 반응 텍스트 조회 |
| `POST /histories` | 선택지 8개 + 학교 ID(건너뛰기 시 0) 제출 → 계산·저장 후 결과 반환 |
| `GET /histories/{id}` | 결과 상세 조회 (공유 링크 재진입) |
| `GET /rankings` | 전체 유형 순위·분포 |
| `GET /schools/{schoolId}/rankings` | 학교별 유형 분포·참여자 수 |

## 9. 앱인토스 심사·정책 체크리스트

- **iframe 사용 금지** (YouTube 삽입 제외) — 심사 반려 사유.
- **외부 앱 설치 유도 금지** — 공유 흐름은 토스 공유 링크(`getTossShareLink`)로만.
- 공포 콘텐츠 수위: 잔혹·혐오 표현 없이 "으스스한" 선 유지 (전연령 기준).
- 학교 정보는 개인 식별 정보가 아니지만, 통계 표본이 적은 학교에서 개인 추정이 가능하지 않도록 최소 표본(예: N명 미만이면 분포 비공개) 정책 검토. ⚠️ 백엔드와 협의.

## 11. 오픈 이슈

### 남은 결정 사항
1. 서비스 정식 명칭 (현재 가칭 "귀신 유형 테스트") — 콘솔 등록 전까지 확정 필요
2. 소표본 학교 통계 비공개 기준 (최소 N명) — 백엔드와 논의 필요
3. BGM/효과음 — MVP 제외, 추후 결정

### 확정된 사항 (본문 반영 완료)
- 학교 선택 건너뛰기 **허용** (제출 시 `schoolId: 0`) → §4, §5.2
- 학교 데이터는 공공데이터 기반으로 **백엔드가 제공** → §5.2, §8
- 학교 간 랭킹 **MVP 제외** (우리 학교 분포는 유지) → §5.5, §6.4
- 선택 후 반응 텍스트(reactionText) **채택** → §5.3
- 일러스트·OG 이미지 16종 **디자인 팀 전달 완료** → §6.1, §7
- 일정: **7/19 프론트 MVP 최소 완료, 7/21 배포** → §10
